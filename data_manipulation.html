Data Manipulation with dplyr

Once you’ve imported data, you’re going to need to do some cleaning up.

Example

Start by figuring out how this example fits in with what you’ve done and plan to do – you may want to start a whole new directory and R Project, but I’d suggest adding a new file to the Project we started last time.

Once again we’re going to be using the tidyverse, so we’ll load that at the outset. We’re going to be looking at a lot of output, so we’ll print only three lines of each tibble by default. Lastly, we’ll focus on the “litters” dataset, so we’ll load those data and clean up the column names using what we learned in data import.

select

For a given analysis, you may only need a subset of the columns in a data table; extracting only what you need can helpfully de-clutter, especially when you have large datasets. You can select columns using select.

You can specify the columns you want to keep by naming all of them:

select(litters_data, group, litter_number, sex, pd_walk)
## # A tibble: 313 x 4
##   group litter_number   sex pd_walk
##   <chr>         <chr> <int>   <int>
## 1  Con7            85     1      11
## 2  Con7            85     1      12
## 3  Con7      1/2/95/2     1       9
## # ... with 310 more rows

You can specify the specify a range of columns to keep:

select(litters_data, group:pd_walk)
## # A tibble: 313 x 6
##   group litter_number   sex gd_of_dob pd_ears pd_walk
##   <chr>         <chr> <int>     <int>   <int>   <int>
## 1  Con7            85     1        20       4      11
## 2  Con7            85     1        20       4      12
## 3  Con7      1/2/95/2     1        19       5       9
## # ... with 310 more rows

You can also specify columns you’d like to remove:

select(litters_data, -pd_ears)
## # A tibble: 313 x 5
##   group litter_number   sex gd_of_dob pd_walk

select(litters_data, GROUP = group, LiTtEr_NuMbEr = litter_number)

If all you want to do is rename something, you can use rename instead of select. This will rename the variables you care about, and keep everything else:

rename(litters_data, GROUP = group, LiTtEr_NuMbEr = litter_number)

##   GROUP LiTtEr_NuMbEr   sex gd_of_dob pd_ears pd_walk

There are some handy helper functions for select; read about all of them using ?select_helpers. The one I use most frequently is everything(), which is handy for reorganizing columns without discarding anything:

select(litters_data, litter_number, sex, everything())
## # A tibble: 313 x 6
##   litter_number   sex group gd_of_dob pd_ears pd_walk
##           <chr> <int> <chr>     <int>   <int>   <int>
## 1            85     1  Con7        20       4      11
## 2            85     1  Con7        20       4      12
## 3      1/2/95/2     1  Con7        19       5       9
## # ... with 310 more rows

Lastly, like other functions in dplyr, select will export a dataframe even if you only select one column. Mostly this is fine, but sometimes you want the vector stored in the column. To pull a single variable, use pull.

filter

Some data tables will include rows you don’t need for your current analysis. Although you could remove specific row numbers using base R, you shouldn’t – this might break if the raw data are updated, and the thought process isn’t transparent. Instead, you should filter rows based on logical expressions using the filter function. Like select, the first argument to filter is the dataframe you’re filtering; all subsequent arguments are logical expressions.

You will often filter using comparison operators (>, >=, <, <=, ==, and !=). You may also use %in% to detect if values appear in a set, and is.na() to find missing values. The results of comparisons are logical – the statement is TRUE or FALSE depending on the values you compare – and can be combined with other comparisons using the logical operators & and |, or negated using !.

Some ways you might filter the litters data are:

  • gd_of_dob == 20
  • pd_walk <= 10
  • pd_ears != 4
  • !(pd_ears == 4)
  • group %in% c("Con7", "Con8")
  • sex == "male" & pd_walk == 9
  • !is.na(pd_ears)

Filtering can be helpful for limiting a dataset to only those observations needed for an analysis. However, I recommend against the creation of many data subsets (e.g. one for male pups and one for female pups). This can clutter up your workspace, and we’ll see good tools for the analysis of subsets before long.

mutate

Sometimes you need to select columns; sometimes you need to change them or create new ones. You can do this using mutate.

The example below creates a new variable measuring the delay between pd_ears and pd_walk; modifies the existing group variable; and creates a new sex variable that takes character values.

mutate(litters_data,
  delay = pd_walk - pd_ears,
  group = tolower(group),
  sex_ch = "male",
  sex_ch = replace(sex_ch, sex == 2, "female")
)
## # A tibble: 313 x 8
##   group litter_number   sex gd_of_dob pd_ears pd_walk delay sex_ch
##   <chr>         <chr> <int>     <int>   <int>   <int> <int>  <chr>
## 1  con7            85     1        20       4      11     7   male
## 2  con7            85     1        20       4      12     8   male
## 3  con7      1/2/95/2     1        19       5       9     4   male
## # ... with 310 more rows

A few things in this example are worth noting:

  • Your new variables can be functions of old variables
  • New variables appear at the end of the dataset in the order that they are created
  • You can overwrite old variables
  • You can create a new variable and immediately refer to (or change) it

Creating a new variable that does exactly what you need can be a challenge; the more functions you know about, the easier this gets.

%>%

We’ve seen several commands you will use regularly for data manipulation and cleaning. You will rarely use them in isolation. For example, suppose you want to load the data, clean the column names, remove pd_ears, and create sex_ch. There are a couple of options for this kind of multi-step data manipulation:

  • define intermediate datasets (or overwrite data at each stage)
  • nest function calls

The following is an example of the first option:

litters_data_raw = read_csv("./data/FAS_litters.csv")
## Parsed with column specification:
## cols(
##   Group = col_character(),
##   Litter.Number = col_character(),
##   Sex = col_integer(),
##   GD.of.DOB = col_integer(),
##   PD.ears = col_integer(),
##   PD.walk = col_integer()
## )
litters_data_clean_names = clean_names(litters_data_raw)
litters_data_selected_cols = select(litters_data_clean_names, -pd_ears)
litters_data_with_sex = mutate(litters_data_selected_cols, 
  sex_ch = "male",
  sex_ch = replace(sex_ch, sex == 2, "female"))
litters_data_with_sex
## # A tibble: 313 x 6
##   group litter_number   sex gd_of_dob pd_walk sex_ch
##   <chr>         <chr> <int>     <int>   <int>  <chr>
## 1  Con7            85     1        20      11   male
## 2  Con7            85     1        20      12   male
## 3  Con7      1/2/95/2     1        19       9   male
## # ... with 310 more rows

Below, we try the second option:

litters_data_clean = 
  mutate(
    select(
      clean_names(
        read_csv("./data/FAS_litters.csv")
      ), 
    -pd_ears),
  sex_ch = "male",
  sex_ch = replace(sex_ch, sex == 2, "female"))
## Parsed with column specification:
## cols(
##   Group = col_character(),
##   Litter.Number = col_character(),
##   Sex = col_integer(),
##   GD.of.DOB = col_integer(),
##   PD.ears = col_integer(),
##   PD.walk = col_integer()
## )
litters_data_clean

## 2  Con7            85     1        20      12   male

These are both confusing and bad: the first gets confusing and clutters our workspace, and the second has to be read inside out.

Piping solves this problem. It allows you to turn the nested approach into a sequential chain by passing the result of one function call as an argument to the next function call:

litters_data = read_csv("./data/FAS_litters.csv") %>%
  clean_names() %>%
  select(-pd_ears) %>%
  mutate(sex_ch = "male",
         sex_ch = replace(sex_ch, sex == 2, "female"))
## Parsed with column specification:
## cols(
##   Group = col_character(),
##   Litter.Number = col_character(),
##   Sex = col_integer(),
##   GD.of.DOB = col_integer(),
##   PD.ears = col_integer(),
##   PD.walk = col_integer()
## )
litters_data
## # A tibble: 313 x 6
##   group litter_number   sex gd_of_dob pd_walk sex_ch
##   <chr>         <chr> <int>     <int>   <int>  <chr>
## 1  Con7            85     1        20      11   male
## 2  Con7            85     1        20      12   male
## 3  Con7      1/2/95/2     1        19       9   male
## # ... with 310 more rows

All three approaches result in the same dataset, but the piped commands are by far the most straightforward. The easiest way to read %>% is “then”; the keyboard shortcuts are Ctrl + Shift + M (Windows) and Cmd + Shift + M (Mac).

The functions in dplyr (and much of the tidyverse) are designed to work very well with the pipe operator. By default, the pipe will take the result of one function call and use that as the first argument of the next function call; by design, functions in dplyr will take a tibble as an input and return a tibble as a result. As a consequence, functions in dplyr are easy to connect in a data cleaning chain. You can make this more explicit by using . as a placeholder for the result of the preceding call:

litters_data = read_csv("./data/FAS_litters.csv") %>%
  clean_names(dat = .) %>%
  select(.data = ., -pd_ears) %>%
  mutate(.data = .)
## Parsed with column specification:
## cols(
##   Group = col_character(),
##   Litter.Number = col_character(),
##   Sex = col_integer(),
##   GD.of.DOB = col_integer(),
##   PD.ears = col_integer(),
##   PD.walk = col_integer()
## )

Usually you’ll just elide the first argument and be happy with life, but there are some cases where the placeholder will help. For example, to regress pd_walk on sex, you might use:

litters_data %>%
  lm(pd_walk ~ sex, data = .) %>%
  broom::tidy()
##          term   estimate std.error  statistic       p.value
## 1 (Intercept)  9.5704777 0.2401359 39.8544313 3.119016e-124
## 2         sex -0.0478971 0.1514404 -0.3162769  7.520045e-01

There are limitations to the pipe. You shouldn’t have sequences that are too long, there isn’t a great way to deal with multiple inputs and outputs, and (since it’s not base R) not everyone will know what %>% means. That said, compared to days when R users only had the first two options, life is much better!
